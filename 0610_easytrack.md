在 Azure DevOps 的 **EasyTrack** 專案管理模板中，**Issue** 不一定僅限於記錄問題或風險，也可以作為 **Epic** 的細項，用來表示更細粒度的目標或子任務，具體取決於你的專案需求和命名慣例。以下是詳細說明，結合你的要求，進一步優化 **EasyTrack** 的 SOP 和 Wiki 內容，確保簡單明瞭，並明確說明如何使用 **Issue** 作為 **Epic** 的細項，同時保持低學習門檻。

---

### 解答：Issue 是否僅限於問題，或可作為 Epic 的細項？

1. **Azure DevOps 中的 Issue 定義**：
   - 在「Basic」模板中，**Issue** 預設用於記錄問題、風險或障礙（例如「伺服器延遲」或「資源不足」）。
   - 然而，Azure DevOps 允許自訂工作項目的使用方式。你可以將 **Issue** 定義為 **Epic** 的子層級，類似於 Agile 模板中的 **Feature**，用來表示較大的工作單元，但比 **Epic** 更細分，具體但不至於像 **Task** 那樣是可執行的最小單位。
   - 例如，**Epic** 可以是「開發新產品」，**Issue** 作為細項可能是「設計前端介面」或「撰寫行銷企劃」，而 **Task** 則是更具體的「設計登錄頁面」或「撰寫廣告文案」。

2. **將 Issue 作為 Epic 細項的優勢**：
   - **靈活性**：允許非程式專案（如行銷、行政）將 **Issue** 用作中間層級目標，簡化層次結構（Epic > Issue > Task），減少學習成本。
   - **一致性**：無論程式或非程式專案，都能用統一的 **Epic**、**Issue**、**Task** 結構管理。
   - **簡化流程**：專案負責人可以同時創建 **Epic** 和 **Issue**，直接將大目標分解為中層目標，無需額外引入其他工作項目類型（如 Feature）。

3. **如何實現**：
   - 在 Azure DevOps 中，**Issue** 可以透過「父子關係」連結到 **Epic**，作為其子項目。
   - 你可以在創建 **Issue** 時，將其標題和描述設定為目標導向（而非問題導向），例如「Issue: 完成 API 整合」而不是「Issue: 伺服器錯誤」。
   - 透過 Kanban 看板或 Backlog 視圖，清楚顯示 **Epic** 下屬的 **Issue** 和 **Task**。

4. **注意事項**：
   - 若將 **Issue** 用作 **Epic** 的細項，需在 Wiki 和培訓中明確定義其用途，避免團隊成員誤認為 **Issue** 僅限於問題。
   - 建議保留 **Issue** 的一部分功能用於問題管理，同時允許其作為目標細項，以保持靈活性。

#### 目的
利用 **EasyTrack** 模板在 Azure DevOps 中統一管理專案，透過 **Epic**、**Issue** 和 **Task** 實現簡單、靈活的流程。**Issue** 可作為問題/風險或 **Epic** 的細項（中層目標），由專案負責人創建，適用於程式與非程式專案。

#### 角色與責任
1. **專案負責人**
   - 創建專案並設定基本資訊。
   - 同時創建 **Epic**（大目標）與 **Issue**（細項目標或問題）。
   - 審核進度，確保 **Epic** 和 **Issue** 進展順利。
   - 主持回顧會議，更新 Wiki。
2. **團隊負責人**
   - 將 **Issue** 分解為 **Task**，或分配 **Issue**（問題）的解決責任。
   - 每日檢查 Kanban 看板，確保進度。
   - 維護 Wiki 的實務內容。
3. **團隊成員**
   - 執行分配的 **Task** 或解決 **Issue**（問題）。
   - 每日更新狀態。
   - 提供 Wiki 建議。
4. **行政支援**
   - 上傳文件，維護 Wiki 結構。
   - 記錄會議紀錄。

#### 流程
1. **專案初始化**
   - **負責人**：專案負責人
   - **步驟**：
     - 登入 Azure DevOps，創建專案，命名為「EasyTrack_專案名稱_2025」。
     - 選擇「Basic」模板，設為「Private」。
     - 邀請成員並分配角色。
   - **成果**：專案準備就緒。

2. **創建 Epic 與 Issue**
   - **負責人**：專案負責人
   - **步驟**：
     - 在「Boards」點擊「New Work Item」，創建 **Epic**（如「推出新產品」）。
     - 創建 **Issue**，作為 **Epic** 的細項（如「設計前端介面」）或問題（如「預算不足」）。
     - 將 **Issue** 連結到對應 **Epic**（設為父項目）。
     - 設定優先級（高/中/低）和預估時間。
   - **成果**：目標與問題結構清晰。

3. **分解與分配工作**
   - **負責人**：專案負責人、團隊負責人
   - **步驟**：
     - 將 **Issue**（細項）分解為 **Task**（如「設計登錄頁面」）。
     - 為 **Task** 或 **Issue**（問題）分配負責人與截止日期。
   - **成果**：工作分配完成。

4. **進度追蹤**
   - **負責人**：團隊負責人、團隊成員
   - **步驟**：
     - 在「Boards > Kanban」查看看板（「待辦」「進行中」「完成」）。
     - 團隊成員每日更新 **Task** 或 **Issue** 狀態。
     - 專案負責人每週檢查 **Epic** 與 **Issue** 進度。
   - **成果**：進度透明。

5. **問題管理**
   - **負責人**：專案負責人、團隊負責人、團隊成員
   - **步驟**：
     - 發現問題時，創建 **Issue**（如「伺服器錯誤」）。
     - 團隊負責人分配解決責任，專案負責人審核。
   - **成果**：問題有效管理。

6. **報告與總結**
   - **負責人**：專案負責人、行政支援
   - **步驟**：
     - 使用「Dashboards」生成報告，顯示 **Epic** 和 **Issue** 狀態。
     - 行政支援記錄會議紀錄並更新 Wiki。
     - 專案結束後，回顧並更新「經驗教訓」。
   - **成果**：報告與知識存檔。

7. **文件與 Wiki 管理**
   - **負責人**：行政支援、所有成員
   - **步驟**：
     - 上傳文件至 Azure DevOps。
     - 更新 Wiki，記錄 **Epic** 和 **Issue** 相關資訊。
   - **成果**：文件與知識庫完善。

- **成果**：進度即時可見。

## 5. 問題管理
- **負責人**：專案負責人、團隊負責人、團隊成員
- **步驟**：
  1. 發現問題時，在「Boards」創建 **Issue**，描述問題並標記優先級。
  2. 團隊負責人分配解決責任，追蹤進度。
  3. 專案負責人每週審核未解決的 **Issue**。
- **成果**：問題快速解決。

## 6. 報告與總結

  3. 專案結束後，召開回顧會議，更新「經驗教訓」。

# 常見問題

**Q：如何創建 Epic 或 Issue？**  
A：在「Boards」點擊「New Work Item」，選擇「Epic」或「Issue」，填寫標題、描述和優先級。

**Q：Issue 和 Task 的區別？**  
A：**Issue** 用於問題或風險，**Task** 用於具體工作。

**Q：忘了更新狀態怎麼辦？**  
A：立即更新，並通知團隊負責人。

**Q：如何上傳文件？**  
A：由行政支援上傳至存放庫，或在 Wiki 嵌入連結（如 `[文件](路徑)`）。

**Q：需要複雜的 Markdown 語法嗎？**  
A：不需要，簡單標題（`#`）和清單（`1.`）即可。
